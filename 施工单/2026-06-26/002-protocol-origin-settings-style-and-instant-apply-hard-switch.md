## 目标

一次性把 popup 顶栏“站点配置”切换到下面这套最终模型：

```txt
站点配置面板
  有完整样式
  是挂在 popup 顶栏下方的 inline 面板
  不是裸表单

配置交互
  复选框点击后立即生效
  数字输入在完成编辑时立即生效
  不再需要“保存”按钮
  不再显示“已保存”提示

失败处理
  保存失败时回滚到上一个已持久化真值
  显示错误
  不做复杂重试
```

本次是硬切换，不接受：

1. 先补点样式，保存按钮先留着；
2. 先把复选框做成即时生效，数字输入以后再收口；
3. 先每敲一个字符都写库，后面再处理输入中间态；
4. 先只做“视觉像样”，不处理失败回滚。

## 简述缘由

1. 当前“站点配置”不是 CSS 没加载，而是根本没有对应样式。`OriginSettingsTray.tsx`
   里用了大量 `origin-settings-panel*` 类名，但 `styles.css` 里没有这些规则，
   所以现在显示出来的就是一块裸表单。

2. 这块配置本质上是 per-origin 策略真值，不是“填一页表单最后统一提交”的业务流程。用户改一个开关，本质上就是在改未来策略，最合理的交互就是立即生效。

3. 但“立即生效”不等于“每个输入字符都落库”。数字输入天然有中间态，例如空串、删到一半、临时非法值。如果把编辑过程当最终真值，会把脏状态持久化。

4. 因此最简单、最稳的做法是：
   - 复选框：点击即写库；
   - 数字输入：在 `blur` 或 `Enter` 这类“编辑完成”时写库；
   - 写失败立即回滚，不做复杂暂存和重试队列。

5. 这符合当前项目一贯原则：优先系统简单，不为了“体验补丁”引入复杂状态机。

## 本单补充定义

### 一、当前问题定义为“缺样式”，不是“样式链路失效”

本次先把事实钉死：

```txt
OriginSettingsTray 的 DOM 结构已经存在
styles.css 已经被 web 入口加载
但 origin-settings-panel 相关类没有任何 CSS 规则
所以展示结果是裸表单
```

这意味着：

1. 本次修复不应该去怀疑 `@import "@keymaster/plugin-protocol/styles.css"` 链路。
2. 也不应该引入新的独立样式文件或在页面里内联 style 兜底。
3. 正确修法就是在现有 `packages/plugin-protocol/src/styles.css` 里把缺失的面板样式补完整。

### 二、站点配置改为“修改立即生效”

本次统一采用下面的交互模型：

```txt
复选框
  点击
  -> 立即写 origin 配置
  -> 成功则保持
  -> 失败则回滚

数字输入
  onChange 只更新本地编辑值
  onBlur / Enter 提交
  -> 立即写 origin 配置
  -> 成功则更新真值
  -> 失败则回滚
```

明确废弃：

```txt
编辑多个字段
最后点“保存”
```

### 三、复选框与数字输入使用不同的即时提交语义

这里必须区分：

#### 复选框

复选框没有中间态，所以：

```txt
click = 最终决定
```

必须立即提交，不需要额外确认。

#### 数字输入

数字输入有中间态，所以：

```txt
键入过程 != 最终决定
```

因此：

- `onChange` 只更新本地编辑态；
- `blur` 或 `Enter` 才提交；
- `Escape` 可选地回退到上次真值，但不是硬要求；
- 不允许“每个字符都写库”。

### 四、失败时必须回滚到上一个已持久化真值

本次把失败语义钉死：

```txt
写库失败
  = 当前修改未生效
  = UI 回滚到上一个已持久化真值
```

不允许：

1. 保存失败了但 UI 还停留在新值，制造“看起来开了，其实没开”的假象；
2. 保存失败后保留一个“脏编辑态”长期挂在界面里，等用户猜自己到底改没改成功；
3. 背景自动重试并悄悄把真值改掉。

### 五、即时生效不新增复杂事务模型

本次不做：

- 多字段批量提交；
- 脏状态提示；
- 撤销栈；
- 乐观更新队列；
- 离线缓存；
- 自动重试策略。

理由：

1. 站点配置的字段数量虽然多，但每一项都可以独立理解与独立提交。
2. 失败时回滚已经足够收口，不值得再加事务层。

### 六、面板视觉定位固定为“顶栏下挂的 inline 配置层”

本次面板定位固定为：

```txt
挂在 protocol popup 顶栏下方
宽度受 popup 本身限制
视觉上属于当前窗口的一部分
不是 modal
不是 drawer
不是整页 settings
```

要求：

1. 打开面板后，当前命令流仍然在同一页面内。
2. 面板视觉上要明显区别于 feed 卡片，但不应喧宾夺主。
3. 关闭方式仍然保持当前“点 × 或再点一次顶栏按钮”。

## 硬切换结论

### 一、移除保存按钮与“已保存”提示

本次直接删除：

- “保存”按钮
- “已保存”提示

因为这两个元素属于旧的“表单集中提交”模型。保留它们只会让“即时生效”语义变得含糊。

### 二、错误提示保留，但语义从“保存整页失败”改成“本次修改失败”

错误区块继续保留，但语义改成：

```txt
刚才这个字段修改没写进去
```

不再是：

```txt
整页表单保存失败
```

### 三、站点配置样式补全在现有 `styles.css` 内完成

不新增：

- `originSettings.css`
- 组件内联 style
- web 侧专门的样式覆盖文件

这样做的原因：

1. 这本来就是 `plugin-protocol` 自己的 UI。
2. 样式真值应该跟着组件一起收口。

### 四、即时生效的真值仍然只落 `origins` store

本次仍然不新增任何新存储层。

即时生效只是把“什么时候调用 `service.setOriginSettings`”改了，不是改持久化模型。

## 不能怎么做

1. 不能把“即时生效”等价成“删除保存按钮，onChange 里每次都写库”。尤其数字输入不能这么做。

2. 不能因为想快点出效果，就把样式直接写在 JSX 里。这样后面很难维护，也和现有样式组织方式冲突。

3. 不能让复选框点击后先改本地 state，但写失败时不回滚。那会直接伪造错误真值。

4. 不能让数字输入写失败后保留失败值。用户会以为已生效，实际 DB 里还是旧值。

5. 不能为了解决“即时保存时的中间态”而重新引入“保存整页”按钮。这是绕回旧模型，不是修复。

6. 不能因为 DB 不可用，就把面板改成“假编辑模式”。不能让用户能点、能输、但其实永远存不进去还不明确提示。

7. 不能顺手大改 popup 整体视觉风格。此次只补站点配置面板，不借题发挥重做 feed 或顶栏。

## 应该怎么做

### 总体策略

一次性做三层收口：

1. 样式层：补齐 `origin-settings-panel*` 的全部 CSS；
2. 交互层：去掉保存按钮，改成字段级即时提交；
3. 测试层：补“点击立即生效”“失败回滚”“数字输入 blur/Enter 提交”的覆盖。

### 设计要点

#### 1. 复选框改成立即提交

建议模式：

```txt
读到旧 record
点击后构造 next record
先更新 UI
调用 service.setOriginSettings(next)
失败则回滚到旧 record 并显示 error
成功则清 error
```

这里不需要额外 debounce。

#### 2. 数字输入分“编辑态”和“已提交真值”

建议为每个数字字段维护一个字符串编辑值，例如：

```txt
p2pkhAutoApproveMaxInput
feePoolAutoSignMaxInput
feePoolDefaultFundInput
```

原因：

1. 用户输入过程需要允许空串与临时中间态；
2. 真值仍然是 number；
3. 只有提交时才把字符串解析成最终数值。

#### 3. 数字字段提交时的规范化规则

建议固定为：

```txt
空串 / 非整数 / 负数
  -> 0

合法非负整数
  -> 该整数
```

不要做：

- 小数四舍五入；
- 保留上次值但 UI 不说明；
- 抛前端校验异常阻断用户离开输入框。

#### 4. 面板样式应包含的最小集合

至少应补齐：

- `origin-settings-panel`
- `origin-settings-panel__form`
- `origin-settings-panel__field`
- `origin-settings-panel__field` 下 checkbox 与数字输入的布局
- `origin-settings-panel__loading`
- `origin-settings-panel__error`
- `origin-settings-panel__close`
- 如保留状态提示，则补 `origin-settings-panel__saving` 一类轻量反馈

视觉目标：

1. 看起来是 popup 顶栏下展开的一块配置层；
2. 字段有明确间距和边界；
3. 移动端不挤爆；
4. 不需要夸张设计，但不能再是裸 HTML。

#### 5. 允许“保存中”轻反馈，但不恢复保存按钮

本次如果要做反馈，只允许做轻量状态，例如：

```txt
字段短暂 disabled
或
小号“保存中…”
```

不允许又做回：

```txt
Save
Saved
```

## 特殊情况提前约定

### 情况 1：复选框点击后写库失败

处理原则：

```txt
回滚
报错
停在旧值
```

说明：

- 不保留新勾选态；
- 不自动重试；
- 用户如果还想改，再点一次。

### 情况 2：数字输入提交时解析失败

处理原则：

```txt
规范化成 0
再提交
```

如果写库成功，UI 显示 `0`。  
如果写库失败，UI 回滚到旧真值。

### 情况 3：数字输入编辑中切换了 origin

处理原则：

```txt
直接丢弃未提交编辑态
重新读取新 origin 真值
```

理由：

- popup 的站点配置永远只对应当前 origin；
- 不值得为未提交编辑再加跨 origin 暂存。

### 情况 4：DB 不可用

处理原则：

```txt
面板仍可打开
任何修改都会失败
UI 显示错误
真值保持不变
```

不额外做只读模式，因为当前 service 已经能通过报错表达“写不进去”。

### 情况 5：连续快速点击复选框

处理原则：

```txt
以最后一次成功写入为准
```

实现上可以用简单 `busy` 门禁：

- 当前字段提交中时临时禁用该字段；
- 不为这类边缘交互引入排队器。

### 情况 6：数字输入 `blur` 与 `Enter` 连续触发

处理原则：

```txt
同一值的重复提交应幂等
```

如果已经提交过当前规范化值，不需要再重复写库。

## 最终验收清单

### 视觉

1. 打开 popup 顶栏“站点配置”后，看到的是一块完整样式的配置面板，不再是裸表单。
2. 面板在桌面端与移动端都能正常阅读，不挤爆、不重叠。
3. 面板视觉上明显属于 popup 顶栏下挂区域，而不是 feed 里的一张普通卡片。

### 交互

4. 复选框点击后立即生效，不需要再点“保存”。
5. 页面上不再出现“保存”按钮。
6. 页面上不再出现“已保存”提示。
7. 数字输入在编辑过程中不会每个字符都触发持久化。
8. 数字输入在 `blur` 或 `Enter` 后立即生效。

### 真值与失败收口

9. 复选框修改成功后，刷新 popup 或重新打开 popup，值仍然保持。
10. 复选框修改失败时，UI 会回滚到旧值，并显示错误。
11. 数字输入提交成功后，刷新 popup 或重新打开 popup，值仍然保持。
12. 数字输入提交失败时，UI 会回滚到旧值，并显示错误。
13. 非法数字输入最终会按 `0` 收口，而不是把脏中间态写进 DB。

### 边界行为

14. 切换到另一个 origin 后，面板会显示新 origin 的真实配置，而不是前一个 origin 的编辑残留。
15. DB 不可用时，面板仍可打开，但任何修改都不会伪装成成功。
16. 不会因为即时生效改动而影响 origin 配置已有的持久化模型，真值仍只落 `origins` store。
